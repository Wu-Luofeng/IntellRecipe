Redis Stream 到 RabbitMQ 分发与确认机制笔记

一、整体目标
把秒杀下单事件先可靠写入 Redis Stream，然后由转发者把事件投递到 RabbitMQ。只有确认 RabbitMQ 已接收并路由成功后，才确认 Redis Stream 这条事件已处理，保证最终一致性。

二、分发方式
1. 增加一个转发者组件
   可以在 voucher-service 内部增加后台线程，也可以做成独立 dispatcher 服务。

2. 转发者处理流程
   1）从 Redis Stream 以消费组方式拉取事件
   2）把事件转换为消息体，发送到 RabbitMQ 的交换机
   3）等待 RabbitMQ 的确认结果
   4）确认成功后，才对 Redis Stream 中对应事件做确认
   5）确认失败则不确认 Redis Stream，留待重试

三、如何判断 broker 有没有收到消息
RabbitMQ 生产者侧使用 Publisher Confirm 确认机制
1. broker 接收并处理完成后，返回 ack
2. 出错或未接收成功，返回 nack
生产者根据 ack 或 nack 判断这条消息是否发送成功。

四、如何判断消息是否成功路由到队列
RabbitMQ 使用 Return 机制
1. 当交换机存在但路由不到队列时，会触发 return 回调
2. 用于发现 routing key 配错、队列未绑定等问题
Return 用来确认路由成功，Confirm 用来确认 broker 接收成功。

五、Redis Stream 与 RabbitMQ 确认串联原则
1. RabbitMQ 未确认成功时，不确认 Redis Stream
2. 让 Redis Stream 事件保留在 pending，后续重试转发
3. 即使发生重复转发，也要靠下游消费端幂等去重保证不重复下单
